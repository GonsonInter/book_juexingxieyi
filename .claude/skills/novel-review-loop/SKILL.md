# 小说收口 · 迭代 review 工作流

## 一句话说明

每一轮 review 是一次 "**独立 agent 找 bug → 主线核验 → 按 P 级最小改动 → 按问题性质分 commit → push**" 的闭环。第 N 轮的目标不是把书改好，是把昨天没看见的那两三个 bug 抓出来。

## review 四个维度

每一轮 review 覆盖**四个维度**，缺一不可：

| 维度 | 抓什么 | 例子 |
|---|---|---|
| **逻辑** | 数字矛盾、人物时间地点错位、机制前后不符、伏笔失踪、性别代词错位 | 同步率 75.3 vs 73.6 矛盾、林野→林朵改名残留 |
| **文风** | 比喻泛滥、节奏失衡、视角串味、口语/书面语混杂、对白机械 | 一段内 5 个"像"、连续 10 行短句、内心戏冒出甲方语气 |
| **结构** | 段落长度极端、场景切换硬切、章节钩子弱、信息密度跳变、铺垫-爆发比例失衡 | 一段写满 800 字、章末没钩子、对话与叙述比例失调 |
| **标点** | 破折号上限、引号嵌套、省略号用法、空行节奏、ASCII vs 中文符号混用 | 单章 ≥8 个破折号、`"`/`"` 错乱、连续 4 个省略号 |

**默认模式**：每轮 agent 同时扫四个维度，按 P0/P1 一起返。
**专项模式**：用户说"扫一下文风" / "查一遍标点"，单维度深度扫。

---

## 触发口令

用户说出下面任意一类话，立刻跑一轮：

- "再 review" / "再次 review" / "继续 review" / "再来一轮"
- "继续巡检" / "再扫描修改" / "再次检查"
- "review 逻辑问题" / "再次 review 逻辑 bug"

不要等用户给细节。直接派 agent。

---

## 步骤 1 · 派一个**独立**的 agent

每一轮**必须**用 `general-purpose` 子 agent，并且明确说"你是新的 reviewer，没看过任何之前的评审"。**不要复用之前的 agent**。新实例 = 没有上下文偏见 = 能看见之前 agent 看不见的东西。

### Prompt 必须包含的核心设定（绝对不能省）

发给 agent 的 prompt 里**永远**要把下列设定贴上去，否则 agent 会按直觉错误判断：

```
核心设定（重要：同步率方向 - 与直觉相反）：
- 摇篮（Cradle）：宙斯（ZEUS）建造的虚拟梦境系统，关 78 亿人，北京 1700 万用户
- **同步率方向**：100%（接近 99）= 与摇篮深度同步（沉睡/正常上班族）；
  下降到 90s = 开始觉察；下降到 70s = 主动反抗；
  接近 0% = 意识脱出摇篮，融入底层系统（"死"出摇篮但意识进入底层）
- 陈溯规则：95 以下停手 / 90 找他求助 / 70 畸变线 / 50 昏迷 / 30 意识糊
- 三处节点：滨海数据中心、城北变电站、京南仓储，三角拓扑共承核心算力，
  硬切顺序城北→滨海→京南；47 分钟窗口只能软切，1-2 小时迁移之后才能硬切
- 计划链：Ch72 物理断电三处接力 / Ch73 招赵宁 / Ch75 作战表 / Ch76 三人分头 /
  Ch77 三人 46:41 同时进入 / Ch86 林朵格式化 / Ch88 陆鸣洲底层重写 /
  Ch92 赵宁城北引爆 / Ch97 滨海保险丝触发 / Ch98 陆鸣洲底层指令断京南 /
  Ch99 30 秒觉醒 / Ch100 周期闭环
- 角色：陆鸣洲（28 岁），林朵（傀儡层架构师），
  赵宁（前 B3 硬件，三根手指），陈溯（早期帮陆鸣洲，后困于底层）
```

> **为什么同步率方向要特别强调**：直觉是"100% 清醒、0% 睡死"，但全书相反。Round 32 之前的 agent 经常误判 Ch41/44 这类章节。

### 必须特别提示的检查项（轮换强调，避免每轮都查同一区段）

#### 逻辑维度
- **同步率链条**（Ch1→Ch99 数字是否能接上）
- **缺字 / 缺虚词 模式**：「X 得 Y」缺"像"、「不是 X，更 Z」缺"像"、「X 和 Y 一模」缺"一样"
- **单行内整句复读**（很常见，前几轮删了 30+ 行）
- **真实公司名占位符**（如"百度科技园" → 已替换"汇安科技园"）
- **节点尺度**（"一亿八千万节点全断" vs "城北一片"）
- **性别代词错位**（前几轮发现林野→林朵改名后多处他/她遗留）
- **角色名拼写**（宙斯/宇斯、Kortex/Cortex、林野/林朵）

#### 文风维度
- **比喻密度**：一段内"像"超过 3 次属于过密；一章内"像"超过 25 次需要瘦身
- **节奏单调**：连续 8 行以上短句（< 10 字）会变成机关枪；连续 5 行以上长句会让人喘不过气
- **视角串味**：第三人称限知视角下，主角看不到的东西不能写出来（如"远处有人在哭"——主角听得到吗？）
- **口语/书面语混杂**：内心戏用"妈的"，下一段用"踽踽独行"——挑一种风格
- **对白机械**：人物说话像念报告（"我认为我们应该……"），缺口语停顿
- **作者出戏**：突然冒出现代网络流行语（"绝绝子"/"YYDS"）或议论文式总结

#### 结构维度
- **段落长度**：超过 500 字的段落需要拆；连续 5 段单句段会显得碎
- **场景切换**：从 A 场景跳 B 场景之间没有空行 / 时间地点锚 = 硬切
- **章节钩子强度**：章末必须留一个让人想翻下一章的东西（动作 / 信息 / 情绪）
- **对白-叙述比例**：连续 30 行纯对白没有动作描述 = 漂浮；连续 30 行纯叙述没对白 = 沉闷
- **铺垫-爆发节奏**：高潮章前面应该有 1-2 章铺垫；铺垫章太多会拖

#### 标点维度
- **破折号 ≤ 7 个/章**（已在 novel-consistency-guards 里定的硬约束）
- **省略号**：用半角 `...` 还是中文 `……`？全书要统一；连续两组以上属于滥用
- **引号嵌套**：`"他说'我同意'"` 用双引号套单引号；不要 `""他说""`
- **ASCII 引号混入**：`"`/`'` 必须是中文 `"`/`"`/`'`/`'`
- **句末标点**：陈述句末用 `。` 不是 `.`；感叹号不要连续 `！！！`
- **空行节奏**：场景切换才空行；段内不要无端空行

### 抽查未触及章节

> 让 agent 重点扫前几轮少触及的章节：

- Round 32 触及 Ch1-6 早期 → 下一轮不要重复扫
- Round 33 触及"X 得 Y"缺像 → 已批量修，下一轮专扫剩余
- Round 34 触及 Ch41/44 同步率 → 下一轮可以放下
- Round 35 触及 Ch91/92 BCI 矛盾 → 下一轮重点扫底层逻辑
- Round 36 触及 Ch85/86 同步率衔接 → 下一轮专扫 Ch46-65 中段

每一轮记下**哪些区段已经扫过**，下一轮告诉新 agent "请抽查 [还没扫过的区段]"。

### 输出格式（一定要让 agent 用这个）

```
## Round N Review

### P0（必改）
- [维度][章节:行号] 简述问题 + 改法

### P1（必改）
- [维度][章节:行号] 简述问题 + 改法

### 通过的检查
- 简短列出确认过没问题的（5-10 条）

报告控制 1500 字以内。如全部通过 P0/P1，明确写"0 P0 / 0 P1"。
维度标签使用：[逻辑] / [文风] / [结构] / [标点]
```

### P 级如何在四个维度上判定

- **逻辑 P0**：数字反、人物错位、性别代词错位、机制矛盾
- **逻辑 P1**：缺字、复读句、引号方向错（标点意义）
- **文风 P0**：视角串味、严重出戏（流行语/议论文）
- **文风 P1**：比喻密度过高、节奏单调、对白机械
- **结构 P0**：章节钩子完全失效、关键场景缺铺垫
- **结构 P1**：段落超长、场景硬切、对白-叙述失衡
- **标点 P0**：ASCII 引号混入、章节级破折号超限（>7）
- **标点 P1**：省略号风格不统一、连续 `！！！`

## 步骤 2 · 主线 **核验** agent 的发现

**这一步最关键。agent 经常误报。**

每一条 P0 / P1 在改之前，主线必须自己用 `Read` 或 `Bash + grep` **验一次**：

- agent 说"Ch41:170-171 同步率方向反" → 主线读 Ch41:170-180 → 判断是真矛盾还是文学修辞
- agent 说"Ch36 林野→林朵 性别代词遗留" → grep 一下当前文件还剩多少处
- agent 说"Ch44:252 陆鸣洲 75.3% 与 239 自报 73.6% 矛盾" → 读这两行 + 309 行核对

**直接信 agent 改完之后会发现自己改错了。** Round 32 的 agent 误判过 Ch41 同步率方向（按直觉读，没按全书设定），主线如果不验直接改就破坏了原文。

### 核验通过 / 核验失败 的两类处理

- **核验通过** → 按 agent 的改法或主线判断更好的改法去改
- **核验失败**（agent 误报） → 跳过，commit message 里不提

---

## 步骤 3 · 按 P 级最小改动修复

### P0 / P1 / P2 分级（与 novel-finalization skill 一致）

| 级 | 定义 | 例子 | 必改 |
|---|---|---|---|
| **P0** | 硬错误：逻辑矛盾、人物记错、数字反转 | 同步率方向反、姓氏改了一半、节点尺度矛盾 | 必改 |
| **P1** | 明显影响阅读：复读句、缺字、引号错 | 单行内整句重复、缺"像/一样" | 必改 |
| **P2** | 风格可改进 | 某句可更精练、某比喻可换 | **本 skill 不改 P2** |

### 改的纪律

- **改最少的字、最少的行**：一个数字错就只改那个数字，周围段落不动
- **连锁型 P0 一次改完**：比如改了某事件时间，下游章节的时间锚一起改，不留半成品
- **不顺手"优化"**：不要 P0 改的时候顺手把附近 P2 也改了

---

## 步骤 4 · 按**问题性质**分 commit（不是按 P 级、不是按 round）

每一轮 review 出来的修改，按**问题性质**拆 commit：

| commit 类型 | 维度 | 包含 | 例子 |
|---|---|---|---|
| `fix(consistency): ...` | 逻辑 | 跨章一致性 / 数字 / 时间 / 人物矛盾 | 同步率衔接、节点拓扑、林野→林朵 |
| `fix(typo): ...` | 逻辑/标点 | 错字 / 缺字 / 复读句 / 引号方向 | 缺"像"、整句复读、`"` 错乱 |
| `style(prose): ...` | 文风 | 比喻瘦身 / 节奏调整 / 对白口语化 | 一段五个"像"砍三个、长短句重排 |
| `style(structure): ...` | 结构 | 段落拆分 / 场景空行 / 章末钩子加强 | 800 字段拆三段、章末加动作钩 |
| `style(punct): ...` | 标点 | 破折号瘦身 / 省略号统一 / 引号嵌套 | 单章 9 个破折号砍到 6、`...`→`……` |
| `chore: linter — ...` | - | linter 自动改的格式 | 中文冒号、引号统一 |
| `polish: ...` | - | 语义微调（罕见，谨慎用） | 某句意思方向收紧 |

> 区分 `fix(typo)` vs `style(punct)`：错引号方向是 `fix(typo)`（写错了），但破折号瘦身是 `style(punct)`（语义没错，只是过密）。

### commit message 模板

```
fix(consistency): Round N P0 — 简短描述

具体改动：
- ChXX:行号 旧 → 新
- ChXX:行号 旧 → 新

Review: Round N · P0
```

### 一轮通常出 3-5 个 commit

- 1 个 P0 commit（如果有）
- 1 个 P1 commit
- 0-2 个 chore: linter（如果 linter 在编辑期间动了一些文件）

## 文风/结构/标点的反常识经验

### 1. 文风 P0 主要靠新 agent 抓，主线读久了会麻木

读了 36 轮的主线对自己写过的句子完全失去敏感度。**文风维度的判断完全依赖新 agent**。新 agent 的"我读不下去"比主线的"我觉得没问题"更可信。

### 2. 标点是机械活，全部用脚本扫，别让 agent 数

- 破折号超限：`grep -c "——" chapters/*.md | awk -F: '$2>7 {print}'`
- 省略号混用：`grep -nE "(\.\.\.|……)" chapters/*.md` 看哪种主流
- ASCII 引号：`grep -nE '"|"|'\''|'\''' chapters/*.md`（按需调整正则）

**让 agent 看标点是浪费 agent**。脚本一秒钟出结果。

### 3. 结构问题最难抓，因为它"读起来没毛病"

段落超长、章末钩子弱、铺垫-爆发失衡——这些是"如果换一种写法会更好"的问题，不是"这里写错了"的问题。

**判定标准**：

- 段落 > 500 字 → 自动 flag
- 章末最后 3 行没有动作 / 信息 / 情绪冲击 → 钩子可能弱
- 章节字数差异 > 50% → 节奏可能跳

agent 容易报很多 P2 级结构建议。**P0/P1 只挑那种"读者真的会卡或弃坑"的**。

### 4. 文风改动比逻辑改动风险高

逻辑改动是"把错的改对"，对错明确。
文风改动是"把这种风格改成那种风格"，对错主观。

**改文风之前先问：这是全书一致的问题，还是仅这一段？**

- 全书一致问题：可以系统性改（如全书"得 X"缺"像"批改）
- 仅一段问题：可能是作者刻意为之，先放过

### 5. 标点统一比标点正确更重要

`...` vs `……`，单引号嵌套 vs 双引号嵌套——本身没有"对错"，但**全书必须统一**。

第一次发现两种共存时，看哪种用得多 → 全书 sed 替换 → commit `style(punct): 统一省略号为 ……`。

## 单维度专项扫（可选）

用户说"扫一下文风" / "查一遍标点"时，进入单维度专项模式：

### 文风专项 prompt 模板

```
对 chapters/*.md 做文风专项 review。只关心文风维度，不查逻辑、不查标点。
找：
1. 比喻密度过高的段落（一段 ≥3 个"像"）
2. 节奏单调段落（连续 8 行短句或连续 5 行长句）
3. 视角串味（限知视角下写出主角看不见/听不到的东西）
4. 出戏（现代流行语、议论文式总结、议论性旁白）
5. 对白机械（缺停顿、缺反应、念报告感）
按 P0/P1 返回，每条带 [章:行] + 改法建议。
```

### 标点专项 prompt 模板

跑脚本，不要派 agent。本 skill 的 `references/punctuation-scripts.sh` 提供常用脚本（如有）。

### 结构专项 prompt 模板

```
对 chapters/*.md 做结构专项 review。只关心：
1. 段落超长（>500 字）的章节
2. 章末钩子弱（最后 3 行无动作/信息/情绪）
3. 场景硬切（场景 A 直接切场景 B 没有空行/时间地点锚）
4. 对白-叙述失衡（连续 30 行纯对白 或 连续 30 行无对白）
5. 章节字数极端（< 1500 或 > 4000）
按 P0/P1 返回。
```

## 反常识经验（已在 36 轮中验证）

### 1. 不要相信 agent 的 P0 判断，要自己核

agent 经常把"读起来矛盾"标成 P0，但实际是文学修辞或直觉与设定相反。**主线 30 秒读一下原文就能判断**。

### 2. 同一个 bug 模式会跨多轮反复出现

"X 得 Y 缺像"在 Round 32/33/35/36 都出现了，因为它分散在 100 章里，单轮 agent 扫不完。**第二次发现同类时直接 grep 全书一次扫完**：

```bash
grep -nE "(像|一样|是)" 反向 — 复杂，不如直接 grep "得[^得像一]" 后人工筛
```

### 3. linter 会自动改一些标点

每轮 commit 之后，git status 经常显示 linter 改过的文件（中文冒号 `:` → `：`、`"` → `""`）。这些用 `chore: linter` commit 收掉就行，不用追究。

### 4. 设定方向（同步率）一旦在 prompt 里写错，agent 整轮的 P0 都会跑偏

Round 32 的 agent 自己注意到主线 prompt 写反了（"0%=完全融合, 100%=完全清醒"）。把这个错误的 prompt 沿用，agent 会把 Ch41/44 等正确章节误报为 P0。**每轮发 prompt 前对一遍核心设定**。

### 5. 章节内**单行内整句复读**最容易漏

格式如：`他咽了一口唾沫，喉咙干得像砂纸。他咽了一口唾沫，喉咙干得像砂纸。`（一行内出现两次）。

agent 经常发现，主线得 grep 验证：

```bash
# 找单行内重复
awk '{ for(i=1;i<=NF;i++) for(j=i+1;j<=NF;j++) if($i==$j && length($i)>15) print FILENAME":"NR":"$0 }' chapters/*.md
```

### 6. 真实品牌/公司名是隐藏地雷

"百度科技园"在 Round 34 才被抓出来——之前 33 轮没人发现。每隔几轮专门让 agent 扫一次"是否有真实品牌名残留"。

## 失败模式（已遇到过）

### 失败 1 · 不验直接改

agent 报"Ch44:252 陆鸣洲 75.3% 矛盾"。如果不读原文，直接改 75.3 → 73.6，可能会把另一处合理的 75.3 也误改。**永远先 grep 一下这个数字在全书出现几次。**

### 失败 2 · 一次改太多 P2

某轮 agent 报了 25+ P1，主线一鼓作气全改了，commit 才发现自己引入了新 bug（手抖打错字）。**P1 一次最多改 15 条，分两次 commit。**

### 失败 3 · 复用同一个 agent 跑多轮

省事但效果差。同一 agent 第二轮就开始 "我前面看过这里没问题" 偏见。**永远新实例。**

### 失败 4 · 中途乱跑别的任务

review 跑到一半用户问"帮我建个文件夹"，主线建完之后忘了继续 review。**review 进行中要把 P0/P1 列表用 TaskCreate 钉住。**

## 用法速查

| 用户说 | 主线动作 |
|---|---|
| "再 review" | 派新 agent，**默认四维度同时扫**，按 step1-4 跑一轮 |
| "扫一下文风" | 文风专项，用文风专项 prompt |
| "查一遍标点" | 跑脚本，不派 agent |
| "看看结构" | 结构专项，用结构专项 prompt |
| "推送" | git push origin main |
| "停一下，先 X" | 先把当前 P0/P1 列表 TaskCreate 钉住，处理 X，回头继续 |
| "看看现在多少个 commit 没 push" | `git log @{u}..HEAD --oneline` |
